# RailMate 功能规格说明书 / 用户故事

## 1. 产品目标

RailMate 不是单一查票购票工具，而是“铁路出行 AI 助手”。它围绕用户从出发前到到站后的关键节点，提供语音输入、自然语言理解、路线规划、票务系统、位置感知、行程管理和个性化推荐能力。

## 2. 目标角色

### 2.1 核心用户

- **高频通勤用户**：周末往返、异地工作、需要快速判断最快/最便宜线路。
- **学生用户**：寒暑假返乡、预算敏感、经常需要多方案比较。
- **旅游用户**：希望顺路玩、看风景、探索中转城市。
- **商务用户**：时间敏感，希望一键得到最稳、最快、最省心方案。

### 2.2 平台角色

- 普通用户
- 登录 12306 的绑定用户
- 后台管理员 / 运营人员

## 3. 功能清单

### 3.1 智能对话能力

- 文本输入查票
- 语音输入转文字
- 自然语言解析日期、城市、时间偏好
- 多轮对话记忆
- 用户位置自动补全出发站
- 偏好驱动的方案推荐

### 3.2 路线规划能力

- 直达查票
- 最快方案推荐
- 最便宜方案推荐
- 最近一班可赶上的车
- 中转方案规划
- 沿途游玩 / 顺路停留方案
- 铁路运转体验模式
- 买长乘短 / 分段购票风险识别与策略建议

### 3.3 票务与账号能力

- 12306 扫码登录
- 票价查询
- 演示购票
- 演示退票
- 行程单查看

### 3.4 用户上下文能力

- IP 自动定位
- GPS 定位
- 手动设置城市/车站
- 用户偏好设置
- 常用线路沉淀

### 3.5 管理与运维能力

- 数据同步
- 任务调度
- 统计接口
- 异步聊天任务

## 4. 核心场景列表

以下场景建议在比赛答辩中按“高频—惊艳—商业化”顺序呈现。

### 场景 A：语音输入

- 用户在路上不方便打字，点击麦克风说：“帮我查下明天广州到长沙最早的高铁。”
- 系统完成语音识别、文本规范化、时间词抽取、站名映射与结果返回。
- 输出内容包括自然语言答复、推荐车次卡片和快速操作按钮。

### 场景 B：自然语言解析

- 用户输入：“我后天下午从景德镇回杭州，便宜一点就行。”
- 系统识别：
  - 出发地：景德镇 / 景德镇北
  - 目的地：杭州 / 杭州东
  - 日期：后天
  - 时段：下午
  - 偏好：价格优先

### 场景 C：位置感知自动补全

- 用户说：“最快到北京的票还有吗？”
- 若用户未给出发地，系统读取当前位置或历史常用出发站作为默认出发地。

### 场景 D：最快 / 最便宜 / 平衡推荐

- 用户分别输入“最快”“最便宜”“差不多就行”。
- 系统按偏好策略自动排序并给出理由。

### 场景 E：中转换乘

- 用户输入：“周六从赣州去青岛，有没有舒服点的中转方案？”
- 系统优先看直达，没有合适直达时再规划一至两次中转，并解释换乘时间与风险。

### 场景 F：沿途游玩 / 顺路停留

- 用户输入：“我想从广州去西安，顺路在长沙玩一天。”
- 系统自动切换到分段行程规划模式，分别生成 `广州→长沙` 与 `长沙→西安` 两段行程。

### 场景 G：买长乘短 / 分段购票

- 用户输入：“上海到南京票太难买，我能不能先买到徐州再提前下车？”
- 系统不直接鼓励高风险操作，而是提供：
  - 规则提示
  - 风险说明
  - 合理替代：分段购票、跨站补票、换站出发、前序站上车建议
- 这一能力非常适合比赛中展示“AI 不只是能回答，更懂规则边界”。

### 场景 H：最近一班赶车

- 用户输入：“我 40 分钟后到广州南，能赶上去深圳的车吗？”
- 系统自动结合当前时间、缓冲时间和站点推荐最近可赶上的列车。

### 场景 I：车次详情与座位价格

- 用户点击某趟 `G` 车次卡片，查看停靠站、耗时、票价、剩余座位。
- 可进一步进入演示购票流程。

### 场景 J：演示购票 / 退票 / 行程管理

- 用户在比赛演示中完成“选座—下单—生成订单—退票—查看行程”全链路。
- 所有订单仅写入本地数据库，不影响真实平台。

### 场景 K：多轮对话纠错

- 用户第一轮：“明天去武汉。”
- 第二轮：“不是武汉，是长沙，越便宜越好。”
- 系统保留上下文并修正意图，不要求用户重说全部条件。

### 场景 L：铁路运转体验模式

- 用户输入：“我不赶时间，想多坐几段车，顺便看看沿线风景。”
- 系统可主动给出更有“乘车体验感”的路线。

## 5. 用户故事

### 5.1 旅客故事

1. **作为一名赶时间的出差用户**，我希望直接告诉系统“最快去北京”，这样我无需手动筛选几十趟车次。
2. **作为一名预算敏感的学生**，我希望系统能自动比较直达和中转的价格差异，以便我省钱回家。
3. **作为一名旅游用户**，我希望系统支持“顺路停留一站”的规划，这样我能把铁路出行和旅行玩法结合起来。
4. **作为一名不方便打字的移动用户**，我希望用语音直接说需求，这样在赶车途中也能快速完成查询。
5. **作为一名对站名不熟悉的用户**，我希望系统能自动把城市转换成推荐高铁站，这样我不容易输错站名。
6. **作为一名不熟悉铁路规则的用户**，我希望系统在买长乘短、补票、中转等场景给我风险提醒，这样我更安心。

### 5.2 运维故事

1. **作为管理员**，我希望能看到数据同步与任务状态，以便保证查询结果稳定可用。
2. **作为产品运营**，我希望能从对话日志中看到高频问题，以便持续优化提示词与产品策略。

## 6. 详细功能规格

## 6.1 智能对话模块

### 输入

- 文本
- 语音转文字
- 快捷操作短语

### 输出

- 自然语言说明
- 车次卡片
- 换乘卡片
- 停留玩法卡片
- 快捷按钮：查看详情、查中转、改日期、演示购票

### 验收标准

- 能识别常见时间词：今天、明天、后天、周末、下午、晚上
- 能识别城市/站名
- 能处理多轮纠错
- 能在无结果时主动补查明天或中转方案

## 6.2 查票模块

### 支持条件

- 出发地
- 目的地
- 日期
- 车次类型筛选
- 时效优先级

### 输出字段

- 车次号
- 车型
- 出发时间 / 到达时间
- 总耗时
- 票价
- 余票
- 推荐标签：最早、最便宜、耗时最短、适合学生、适合出差

## 6.3 中转与沿途规划模块

### 中转规划

- 用户明确提出换乘
- 或直达不足时触发备选方案

### 沿途游玩

- 识别“顺路”“中途玩”“先去 A 再去 B”“停留一天”等表达
- 若停留时长不明确，先追问
- 若明确，生成多段车票建议

## 6.4 买长乘短 / 分段购票策略模块

### 目标

- 给用户“有边界的聪明建议”，而不是盲目鼓励风险行为。

### 关键输出

- 规则提示
- 是否建议执行
- 更合规替代方案
- 成本差异对比

### 比赛价值

- 展示 AI 产品不仅懂查询，还懂规则与风险控制。

## 6.5 定位与上下文模块

- 首次进入自动 IP 定位
- 授权后 GPS 精准定位
- 用户可手动修正城市和车站
- 记忆偏好与常用路线

## 6.6 演示票务模块

- 演示购票
- 演示退票
- 演示订单查询
- 演示模式状态提示

## 7. 优先级规划

### P0：比赛必演示

- 文本/语音输入
- 自然语言查票
- 智能补全出发地
- 最快/最便宜推荐
- 车次卡片 + 详情页
- 演示购票

### P1：拉高惊艳度

- 中转方案
- 沿途游玩
- 12306 二维码登录
- 用户偏好与多轮记忆

### P2：商业化增强

- 买长乘短 / 分段购票策略
- 订阅提醒
- 行程托管
- 本地生活联动推荐

## 9. 总结

RailMate 的功能设计不只是把“票”查出来，而是把铁路用户真正会遇到的决策问题系统化地产品化了：语音输入、自然语言理解、位置感知、分模式规划、沿途游玩、买长乘短风险提醒、演示购票闭环共同构成了一个高频、真实、易展示、可商业化的 AI 场景产品。
