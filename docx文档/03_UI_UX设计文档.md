# RailMate UI / UX 设计文档

## 1. 设计目标

RailMate 的 UI/UX 设计遵循三个核心原则：

- **像聊天一样自然**：让用户先说需求，而不是先填表。
- **像工具一样可靠**：关键车次信息必须卡片化、结构化、可点击。
- **像出行助手一样贴身**：在关键时刻主动给用户下一步行动。

产品采用“双入口设计”：

- `CUI`：对话式输入，负责承接模糊需求、情绪化表达和连续追问。
- `GUI`：卡片式界面，负责承接结构化信息、比较决策和操作执行。

## 2. CUI 对话流设计

## 2.1 标准对话流

```text
用户输入需求
   ↓
系统理解意图
   ↓
补齐缺失信息（如出发地/日期）
   ↓
调用工具查票/查中转/查详情
   ↓
返回自然语言总结 + 卡片
   ↓
用户点击卡片继续查看 / 购票 / 改条件
```

## 2.2 对话流一：直达查票

```text
用户：明天从广州去长沙最快的高铁
系统：已为你查到明天广州南 → 长沙南的直达高铁，
      推荐最早可出发方案如下：
      [车次卡片1] [车次卡片2] [更多]
用户：第二趟多少钱？
系统：二等座 ¥xxx，一等座 ¥xxx，是否查看完整时刻表？
```

### 设计要点

- 首轮回答不堆砌过多数据，只给 2~3 个最优结果。
- 第二轮再展开票价、停靠站、替代方案。

## 2.3 对话流二：语音输入

```text
[用户点击麦克风]
系统状态：正在听...
用户：后天下午回南昌，便宜一点
系统：已识别为“后天下午去南昌，价格优先”。
      如果你默认从景德镇北出发，我现在就帮你查。
      [确认并查询]
```

### 设计要点

- 先展示识别结果，让用户建立信任感。
- 对语义不完整时，用一句确认话术快速补齐。

## 2.4 对话流三：沿途游玩

```text
用户：我想从广州去西安，顺路在长沙玩一天
系统：收到，这是“分段行程 + 沿途停留”场景。
      我会先查 广州→长沙，再查次日 长沙→西安。
      [分段行程卡1]
      [分段行程卡2]
      [长沙停留建议]
```

### 设计要点

- 明确告诉用户系统理解的是“旅行规划”而不是普通换乘。
- 将“交通段”和“停留建议”分开排版。

## 2.5 对话流四：买长乘短 / 风险提醒

```text
用户：上海到南京票没了，买到徐州能不能提前下？
系统：这个场景涉及买长乘短规则，我可以先帮你做风险评估：
      1. 该做法可能涉及检票与席位规则风险
      2. 更推荐的替代方案是分段购票/补票/换站出发
      3. 我可以继续帮你比较三种方案的时间与成本
      [查看替代方案]
```

### 设计要点

- 不直接回答“可以/不可以”，而是给规则感 + 替代感。
- 体现“负责任的 AI”。

## 3. GUI 卡片原型描述

## 3.1 首页布局原型

```text
+----------------------------------------------------------------------------------+
| RailMate Logo | 会话历史 | 新建对话 | 当前位置：景德镇北 | 偏好：平衡 | 设置        |
+----------------------+---------------------------------------------+-------------+
| 最近对话 / 常用路线   |                                             | 用户上下文   |
| - 去长沙              |               对话主区域                    | - 当前城市   |
| - 回广州              |  ┌───────────────────────────────────────┐  | - 推荐车站   |
| - 去北京出差          |  | 用户消息                              |  | - 常用线路   |
|                      |  | AI回答 + 车次卡片 + 快捷操作            |  | - 规划模式   |
|                      |  └───────────────────────────────────────┘  |             |
|                      |  [语音按钮] [输入框....................] [发送] |             |
+----------------------------------------------------------------------------------+
```

### 描述

- 左栏强调历史和快捷路线，帮助高频用户“秒回访”。
- 中间是产品主舞台，卡片和机器人欢迎态都放在这里。
- 右栏用于上下文强化，向评委展示“这不是单轮问答，而是有记忆的 Agent”。

### 描述

- 第一行给用户建立“选择理由”。
- 第二行突出时间与效率。
- 第三行突出价格与购买意图。
- 第四行承接下一步操作。

## 4. GUI 与 CUI 协同原则

### 原则一：对话负责理解，卡片负责落地

- 模糊需求用聊天承接。
- 明确结果用卡片承接。
- 关键操作一定落到按钮，而不是继续让用户口头表达。

### 原则二：每个回答都要有“下一步”

例如：

- 查看详情
- 查中转
- 改日期
- 演示购票
- 加入行程

### 原则三：先结论，后解释

AI 回答应先给“推荐哪一趟 / 为什么”，再给详细说明，否则用户会觉得啰嗦。

## 5. 关键交互细节

## 5.1 语音按钮

- 默认状态为麦克风图标
- 录音中变为脉冲态
- 顶部浮层实时显示“正在听...”或临时转写结果
- 识别失败时显示短错误提示

## 5.2 结果卡片动效

- 新卡片从下方轻微上浮
- 多卡片按 40ms 左右延迟依次出现
- 不使用夸张动画，强调“稳、快、专业”

## 5.3 空结果反馈

空结果不能只说“没有车次”，而要给下一步：

- 改查明天
- 改查中转
- 换附近站点
- 改价格优先 / 时间优先

## 5.4 高风险场景提示

对买长乘短、跨站补票、异常站名、国外定位等情况，要：

- 用黄/橙色提示卡显示风险
- 不打断用户，但提醒规则边界
- 给合规替代操作

## 6. UniApp 适配建议

- 首页保留单列布局，左侧历史与右侧上下文收纳到抽屉中。
- 输入区固定底部，支持语音与快捷短语。
- 结果卡片为纵向滑动列表，更适合移动场景。
- 重要按钮加大点击热区，适应移动端单手操作。

## 7. 结论

RailMate 的 UI/UX 不是传统票务系统的“表单 + 列表”，而是“对话理解 + 卡片决策 + 连续操作”的 AI 原生交互范式。CUI 帮用户表达意图，GUI 帮用户快速比较与执行，两者结合后，产品既有科技感，也有铁路出行场景真正需要的效率感。
